## MiniGrid (Symbolic / One-Hot Observation)

**Summary & Observation/Action Specification**

MiniGrid is a lightweight, discrete-state grid world environment from the Farama Foundation (successor to OpenAI Gym). The agent occupies a cell in an N×N grid with a viewing cone showing walls, doors, keys, and goal tiles as one-hot encoded vectors. The action space is discrete (6 actions: forward, left, right, pickup, drop, toggle). Observations are returned as one-hot tensors of shape (tile_width, tile_height, num_tile_types), making it naturally suited for symbolic world models and causal reasoning but not for deep visual feature learning. The environment is minimal but composable—you can stack rooms, keys, balls, and locks to create multi-step planning tasks.

**VRAM Cost & Wall-Time**

For a Dreamer-V3-small-class model (compact world model with ~2M parameters) running 100k–500k environment steps:
- **Estimated VRAM**: 3–4 GB (world model + value network + replay buffer on GPU). The one-hot observation format is memory-efficient; no image decoding overhead.
- **Wall-time on RTX 4060 Laptop (8 GB total)**: 2–4 hours for 500k steps. The discrete action space and small observation tensors make rollouts fast (200–400 env steps/sec).

**Foresight Metrics Support**

MiniGrid naturally supports:
- **Multi-step image prediction error**: Measure L2 reconstruction error on one-hot tile predictions 1–16 steps ahead.
- **Value rollouts**: Open-loop planning via learned value function; compare predicted vs. actual bootstrap values.
- **Reward forecasting**: Predict whether the agent reaches the goal within N steps.
- Clean ground-truth observation availability (one-hot structure) enables pixel-perfect prediction evaluation with no ambiguity from stochastic rendering.

**Biological Richness Scoring (1–5 scale)**

| Pillar | Score | Notes |
|--------|-------|-------|
| SEEKING | 2 | Goal tile exists but is static; no dynamic resource gradient or exploration reward. |
| FEAR | 1 | No adversaries or danger. Lava tiles exist in some variants but don't move. |
| PLAY | 1 | No interactive dynamics or emergent complexity. Puzzle-solving is not "play" in the instinct sense. |
| Homeostasis | 1 | No hunger, fatigue, or internal state variables exposed in observations. |
| Partial Observability | 4 | Agent has a limited viewing cone (e.g., 7×7); must explore to learn the full map. Excellent foresight driver. |

**Overall Biological Richness**: 2/5. MiniGrid is abstract and goal-centric, lacking the embodied drives needed for a biology-grounded model. Its main strength is partial observability, which forces long-horizon planning.

**License & Maintenance Status**

- **License**: MIT (permissive, commercial use allowed).
- **Maintenance**: Active. Latest release/commit: 2024–2025 (Farama Foundation maintains actively; MiniGrid is part of the Gymnasium ecosystem).
- **Last major update**: 2023–2024 (stable; minor fixes continuing).

**Windows-Specific Gotchas**

- **Rendering**: MiniGrid includes optional pygame-based rendering. On Windows, pygame can be flaky with display servers in WSL2; however, MiniGrid is fully compatible with headless mode (just don't call `.render()`).
- **Installation**: `pip install minigrid` works seamlessly on Windows. No MSVC or external dependencies required.
- **No MuJoCo licensing concerns**: MiniGrid is pure Python.

## MiniGrid with RGB Pixel Observations

**Summary & Observation/Action Specification**

This is the same MiniGrid environment but with observations wrapped to RGB images (typically 64×64 or 128×128 pixels, rendered from the underlying tile map). Action space remains discrete (6 actions). The pixel wrapper discretizes the continuous tile rendering into a visual observation stream, simulating the challenge of learning from camera input. This variant bridges symbolic and pixel-based learning.

**VRAM Cost & Wall-Time**

For Dreamer-V3-small with RGB observations (64×64 RGB) over 100k–500k steps:
- **Estimated VRAM**: 5–6 GB. The image encoder (typically a small CNN or VAE) and stored image replay buffer consume more memory than the symbolic version.
- **Wall-time on RTX 4060 Laptop**: 3–5 hours for 500k steps. Image encoding adds ~30–50% overhead compared to one-hot observations; however, rollouts remain efficient (100–200 env steps/sec).

**Foresight Metrics Support**

- **Multi-step image prediction error**: Measure pixel-level MSE/SSIM on 1–16 step predictions. More challenging than symbolic due to rendering artifacts and pixel-space noise.
- **Value rollouts**: Identical to one-hot variant.
- **Reward forecasting**: Same as one-hot variant.
- Slightly messier ground truth (rendering can introduce anti-aliasing artifacts), but still cleaner than real pixel environments.

**Biological Richness Scoring (1–5 scale)**

Same as symbolic MiniGrid:

| Pillar | Score |
|--------|-------|
| SEEKING | 2 |
| FEAR | 1 |
| PLAY | 1 |
| Homeostasis | 1 |
| Partial Observability | 4 |

**License & Maintenance Status**

Identical to symbolic MiniGrid (MIT, Farama Foundation, actively maintained 2024–2025).

**Windows-Specific Gotchas**

- **Rendering**: RGB wrapper requires pygame for rendering tiles to images. pygame on Windows is stable but can be slow if you're rendering every frame for large observation batches. Headless mode (no rendering) is always available.
- **Image codec**: MiniGrid uses PIL/Pillow for image operations. Installation on Windows is transparent.
- **No additional CUDA/MSVC requirements**: Pure Python + torch.

## Crafter (Hafner)

**Summary & Observation/Action Specification**

Crafter is a 2D top-down crafting and survival environment designed by Hafner (author of Dreamer-V1/V2) as a testbed for world models. The agent navigates a procedurally-generated world, collecting resources (wood, stone, coal), crafting tools, cooking food, and fighting mobs (cows, skeletons, zombies). Observations are RGB images (64×64 pixels). Actions are discrete: move (8 directions), attack, craft (9 recipes), and no-op. The environment includes hunger (gradual health decrease if not fed), enemy spawning, and a crafting skill tree.

**VRAM Cost & Wall-Time**

For Dreamer-V3-small on Crafter over 100k–500k steps:
- **Estimated VRAM**: 5–7 GB. Larger, noisier RGB images (64×64) and more complex dynamics require a bigger world model and longer episode buffers. Crafter episodes run up to 1000 steps (longer than MiniGrid), so replay buffer occupies more space.
- **Wall-time on RTX 4060 Laptop**: 5–8 hours for 500k steps. More complex enemy AI and physics make environment stepping slower (50–100 env steps/sec), and image encoding overhead is higher.

**Foresight Metrics Support**

- **Multi-step image prediction error**: RGB prediction at 1–16 steps; however, enemy movement and stochastic resource spawning make ground truth variable (expect larger, noisier error curves than MiniGrid).
- **Value rollouts**: Useful but less clean due to stochastic enemy behavior (RL agents often learn to avoid enemies rather than predict their motion).
- **Reward forecasting**: Multi-modal: can predict resource discovery, enemy encounters, hunger, and tool completion. More rich than MiniGrid.
- **Skill evaluation**: Can measure learned crafting skill trees and tool progression.

**Biological Richness Scoring (1–5 scale)**

| Pillar | Score | Notes |
|--------|-------|-------|
| SEEKING | 4 | Rich resource collection (wood, stone, coal, food, diamonds). Agent explores to find better biomes and rarer resources. Strong exploratory drive. |
| FEAR | 3 | Enemies (cows, skeletons, zombies) spawn and move toward the agent. Health depletion creates avoidance behavior, but enemies are simple and predictable. Not visceral "fear" but adequate. |
| PLAY | 2 | No emergent play mechanics; gameplay is goal-driven (survive, craft). Some interaction possibility with environment, but limited. |
| Homeostasis | 4 | **Hunger is explicitly tracked**: agent health decreases if not fed. Food is a critical resource. Homeostatic drive is present and central to the game loop. |
| Partial Observability | 3 | Agent sees a local region (implicit from the sprite-based rendering) but does not have a full map. Some exploration needed, but the visual field is relatively generous. |

**Overall Biological Richness**: 3.2/5. Crafter excels in SEEKING (resource collection) and homeostasis (hunger), which are two of the four pillars you requested. FEAR is present but underdeveloped; enemies are slow and telegraphed. PLAY is absent.

**License & Maintenance Status**

- **License**: MIT (permissive).
- **Maintenance**: Moderate. Hafner's original repo (github.com/danijar/crafter) is stable but receives infrequent updates. Last commit: ~2022–2023. Community forks exist (e.g., Crafter-XL with expanded content), but the base environment is effectively in maintenance mode.
- **Ecosystem integration**: Works with Gymnasium and Gym-compatible frameworks.

**Windows-Specific Gotchas**

- **Rendering**: Crafter uses pygame for rendering. Windows support is functional but pygame can be slow on older drivers or WSL2. Headless mode is supported.
- **Dependencies**: Requires Pillow, pygame, numpy. All install cleanly on Windows via pip.
- **No MuJoCo licensing**: Crafter is pure Python and custom physics (very simple collision detection and movement).
- **MSVC**: Not required; pure Python.

## Atari-100k Benchmark (Standard Subset)

**Summary & Observation/Action Specification**

The Atari-100k benchmark consists of ~26 Atari 2600 games (e.g., Pong, Breakout, Space Invaders, Qbert, Montezuma's Revenge) wrapped via ALE (Arcade Learning Environment) and accessed through Gymnasium. Observations are RGB images (84×84 downsampled from 210×160), or optionally stacked grayscale frames. Actions are discrete (typically 4–18 per game). The "100k" refers to a strict evaluation protocol: agents are trained on 100k environment steps (~27 minutes of real gameplay) and then evaluated. This benchmark is the de facto standard for low-data RL in the research community.

**VRAM Cost & Wall-Time**

For Dreamer-V3-small on Atari-100k benchmark over 100k environment steps:
- **Estimated VRAM**: 6–8 GB. Atari games are noisy, visually complex (with flickering enemies, scrolling backgrounds, dynamic text), and require a robust visual encoder. The Dreamer-V3 model for Atari is larger than the MiniGrid variant. Replay buffers must store many episodes to get stable training.
- **Wall-time on RTX 4060 Laptop**: 4–6 hours per game for 100k steps. Atari emulation is fast (500+ steps/sec), but the large visual encoder (typically a ResNet or Vision Transformer module) dominates training time. Expect ~60–120k gradient steps to be computed.

**Foresight Metrics Support**

- **Multi-step image prediction error**: Atari is notoriously hard for pixel-space prediction due to flickering, sprites with 1-pixel jitter, and background scrolling. Prediction error at 1–5 steps is noisy. Multi-step prediction (6+) is extremely difficult and often abandoned in practice.
- **Value rollouts**: Feasible; many Atari games have clear reward signals (score increases). However, value drift due to visual noise is a known problem.
- **Reward forecasting**: Useful for score-based games but dangerous for complex games with delayed rewards (Montezuma's Revenge, Exploration games).
- **Metrics are noisy**: Unlike MiniGrid/Crafter, evaluating foresight on Atari requires filtering or averaging over many seeds; single-run evaluations are unreliable.

**Biological Richness Scoring (1–5 scale)**

Highly game-dependent. Representative scores for a "typical" Atari game (e.g., Breakout):

| Pillar | Score | Notes |
|--------|-------|-------|
| SEEKING | 3 | Agents seek to maximize score. The reward signal (points) is an extrinsic proxy, not intrinsic. Some games have exploration incentives (Montezuma's Revenge), but most are score-chasing. |
| FEAR | 2 | Enemies exist in many games but are not visceral. Collision = game over, but the agent doesn't "learn to fear" in a behavioral sense; it learns to avoid pixels. |
| PLAY | 0 | No emergent play. Pure score-maximization. |
| Homeostasis | 0 | No internal state variables. No hunger, fatigue, or health tracking (except as part of the game's reward structure). |
| Partial Observability | 2 | Full visibility of the game state (all relevant sprites are on screen). Some games have fog-of-war mechanics, but most are fully observable. Poor foresight driver for long-horizon planning. |

**Overall Biological Richness**: 1.4/5. Atari games are reward-maximization tasks, not biology-grounded environments. They lack internal homeostatic drives, emergent play, and meaningful fear/risk. Partial observability is weak.

**License & Maintenance Status**

- **License**: Atari ROM licensing is complex. The **Gymnasium Atari wrapper** itself is MIT, but the ROM files have ambiguous legal status. In practice: (1) Farama distributes pre-converted ROMs in `gymnasium[atari]`, (2) commercial use is generally avoided due to licensing ambiguity, (3) academic research is tolerated.
- **Maintenance**: Active. Gymnasium's Atari integration is actively maintained by Farama (2024–2025). The underlying ALE emulator is stable (last major update ~2020).
- **Ecosystem**: Gymnasium-based, drop-in compatible with standard RL frameworks.

**Windows-Specific Gotchas**

- **Installation**: `pip install gymnasium[atari]` works on Windows. However, it downloads and caches ROM files automatically. Network access required.
- **CMake requirement**: ALE (the emulator) requires CMake to build from source. However, pre-built wheels exist for Windows (MSVC is not required at the user level).
- **ROM storage**: ROMs are cached in `~/.cache/gymnasium/atari/roms`. On Windows, this expands to `%USERPROFILE%\.cache\gymnasium\atari\roms`. Ensure you have ~1 GB free space.
- **Headless rendering**: Atari has optional rendering via pygame; headless mode works fine.
- **No MuJoCo**: Not used.

## dm_control Suite (Small Tasks)

**Summary & Observation/Action Specification**

dm_control is DeepMind's continuous-control benchmark built on the MuJoCo physics engine. Recommended small tasks for low-VRAM training:
- **Cartpole-balance/swingup**: Agent controls a cart's horizontal force to balance a pole.
- **Reacher**: 7-DOF robotic arm reaches a target sphere in 3D space.
- **Finger-spin**: Fingertip must spin a cube at a target angular velocity.
- **Walker-stand**: Bipedal walker maintains upright posture.

Observations are either state vectors (joint angles, velocities, contact sensors: ~10–50 floats) or RGB images (64×64 from an internal renderer). Actions are continuous (box of floats: 1–6D depending on task). Episodes are typically 1000 steps.

**VRAM Cost & Wall-Time**

For Dreamer-V3-small on dm_control state-vector observations over 100k–500k steps:
- **Estimated VRAM (state vector)**: 2–3 GB. Very efficient because observations are small dense vectors, not images. Extremely tight memory footprint.
- **Estimated VRAM (RGB images)**: 4–5 GB. Internal rendering is relatively fast but requires an encoder.
- **Wall-time on RTX 4060 Laptop**:
  - **State vector**: 1–2 hours for 500k steps (very fast; mostly compute-bound on GPU).
  - **RGB images**: 3–4 hours for 500k steps (rendering adds overhead).

**Foresight Metrics Support**

- **Multi-step state prediction**: Excellent. Predicting joint angles 1–16 steps ahead is clean and well-defined (small numerical errors, no ambiguity).
- **Value rollouts**: Excellent. Continuous control has well-behaved value functions; comparing predicted vs. actual bootstrap values is standard practice.
- **Reward forecasting**: Task-dependent. Distance-to-target rewards are easy to predict; penalizing action magnitude and energy is harder.
- **Forward model evaluation**: dm_control is the gold standard for evaluating learned dynamics models because ground truth is deterministic and precise.

**Biological Richness Scoring (1–5 scale)**

| Pillar | Score | Notes |
|--------|-------|-------|
| SEEKING | 3 | Agent seeks to reach targets (spatial goals) or maintain balance (implicit goal). Not resource-driven; goal is externally defined. |
| FEAR | 0 | No adversaries, no harm signals beyond task failure. No predators or danger. |
| PLAY | 1 | No emergent interaction or play behavior. Purely goal-driven control. |
| Homeostasis | 2 | Implicit: agent must maintain balance or posture, which mimics postural homeostasis. However, no explicit hunger, fatigue, or internal state. |
| Partial Observability | 1 | State vector observations give full observability. RGB observations are fully observable (entire scene is visible). Minimal foresight pressure from missing information. |

**Overall Biological Richness**: 1.4/5. dm_control excels at foresight evaluation but fails the biology requirement. There is no fear, play, or meaningful seeking (only externally-imposed goals). Homeostasis is a weak proxy for balance maintenance.

**License & Maintenance Status**

- **License**: Apache 2.0 (permissive, commercial use allowed).
- **Maintenance**: Active. DeepMind maintains dm_control with regular updates (2024–2025). Stable and widely used in RL research.
- **Ecosystem**: PyPI package `dm-control`, compatible with standard gym/gymnasium interfaces (via wrappers).

**Windows-Specific Gotchas**

- **MuJoCo licensing**: MuJoCo was proprietary until 2021, now fully open-source under Apache 2.0. However, **old licensing artifacts**: if your system has an old MuJoCo install, you may need to remove the `~/.mujoco` directory and clear any old license key references.
- **Installation**: `pip install dm-control` works on Windows. Installs MuJoCo binaries automatically.
- **Rendering on Windows**: dm_control uses mjvisualize (native MuJoCo renderer). On Windows with older GPU drivers, rendering can be slow or occasionally flaky. Headless mode (observations only, no rendering) works perfectly and is recommended for training.
- **Headless training**: Use `render=False` and skip any rendering calls; no issues.
- **MSVC**: Not required at the user level; pre-built wheels include MuJoCo binaries.
- **Visual C++ redistributables**: Some users report needing MSVC 2015+ runtime on Windows, but this is rare and typically installed by Windows Update.

## Custom "Instinct Gridworld" (Not Off-the-Shelf)

**Summary & Specification**

A custom 2D top-down grid environment implementing core biological drives:
- **World**: N×N grid (recommend 32×32 or 64×64).
- **Partial observability**: Agent has a 7×7 observation window (centered on agent), perceiving tile types within that cone.
- **Food tiles**: Randomly spawning resources (respawn every K steps). Eating food recovers hunger.
- **Hunger homeostasis**: Agent's hunger stat decreases by 1 every step (0–100 scale). At hunger=0, health decreases. Agent can eat adjacent food to recover hunger. Hunger is visible in the observation vector.
- **Predator(s)**: 1–3 simple predators that move toward the agent using Manhattan distance heuristic. Contact with predator deals damage (decreases health by 10). Fear state (e.g., proximity to predator) can be inferred from predator positions in the observation window.
- **Shelter tiles**: Safe zones that block predator line-of-sight (predators refuse to enter shelter or lose track of agent). Agent can hide to recover and plan.
- **Fatigue homeostasis**: Agent's fatigue increases with movement; high fatigue (>80) reduces max speed. Rest in shelter recovers fatigue.
- **Observation vector**: One-hot tile types + continuous scalars: [agent_x, agent_y, hunger, fatigue, health, predator_proximity_min]. Total ~120–150 dimensions.
- **Action space**: Discrete 6–8 actions (move N/S/E/W, eat, rest, no-op).
- **Reward**: Extrinsic (reach a goal) + intrinsic (shaping: survival bonus, food discovery bonus, predator avoidance penalty). Encourage multi-task reasoning.

**Development Cost Estimate**

For a clean, Gymnasium-compatible implementation reusing MiniGrid patterns:

| Component | Effort | LOC | Notes |
|-----------|--------|-----|-------|
| Grid world core (movement, collision) | 2–4 hours | 150–200 | Trivial; adapt MiniGrid's grid logic. |
| Tile types & rendering | 1–2 hours | 80–120 | Food, wall, shelter, predator types. Copy MiniGrid's tile system. |
| Hunger system | 1–2 hours | 100–150 | Increment/decrement logic, eating mechanics. Standard state tracking. |
| Fatigue system | 1–2 hours | 100–150 | Similar to hunger; affects movement speed. |
| Predator AI | 2–3 hours | 150–250 | Manhattan distance pathfinding, line-of-sight (shelter blocks), simple heuristic. |
| Observation wrapping (partial observability) | 1–2 hours | 80–120 | Extract 7×7 cone around agent; concatenate scalar drives. |
| Gymnasium interface (reset, step, render) | 1–2 hours | 100–150 | Standard gym boilerplate. |
| Testing & debugging | 2–3 hours | — | Ensure hunger/fatigue/predator mechanics work. |

**Total estimate**: 11–20 person-hours (~1.5–2.5 person-days for a single developer). **Total LOC**: ~900–1200 lines of well-organized Python. Feasible for in-house development; lower risk than integrating an unfamiliar third-party environment.

**VRAM Cost & Wall-Time**

For Dreamer-V3-small over 100k–500k steps:
- **Estimated VRAM**: 3–4 GB. Observations are mostly one-hot + small dense vectors (no image encoding needed). Efficient memory profile.
- **Wall-time on RTX 4060 Laptop**: 2–4 hours for 500k steps (similar to symbolic MiniGrid; observation size is comparable). Environment stepping is fast (300–500 steps/sec) because there's no rendering or physics simulation.

**Foresight Metrics Support**

- **Multi-step prediction**: Predict tile positions, predator positions, and continuous drive states 1–16 steps ahead. Clean, deterministic ground truth.
- **Value rollouts**: Predict cumulative reward via learned value function; compare with actual bootstrap. Useful for analyzing which drives the agent learns to pursue.
- **Drive forecasting**: Can explicitly measure whether the agent predicts hunger/fatigue/health, and whether it plans to mitigate them. Excellent for analyzing biology-grounded learning.
- **Behavioral metrics**: Place-cell-like rate maps (agent firing rate as a function of [hunger, location]) can be extracted to probe internal representations.

**Biological Richness Scoring (1–5 scale)**

| Pillar | Score | Notes |
|--------|-------|-------|
| SEEKING | 5 | Food collection is the primary driver. Agent explores to find food tiles, learns spatial locations, and optimizes foraging paths. Strong intrinsic motivation. |
| FEAR | 4 | Predators move toward agent; contact deals damage. Agent learns to avoid, hide, and strategize escapes. Not intense (no "death spiral"), but genuine avoidance behavior. |
| PLAY | 2 | Limited emergent interaction, but shelter-hiding + predator evasion can feel like playful dynamics. Not the main focus. |
| Homeostasis | 5 | **Hunger** and **fatigue** are explicit, continuous, and central to gameplay. Agent must balance exploration (depletes fatigue), foraging (reduces hunger), and rest (homeostatic recovery). Rich internal state. |
| Partial Observability | 5 | 7×7 observation window forces the agent to explore the world, remember visited locations, and plan to visit food spawns it discovered earlier. Strong foresight pressure. |

**Overall Biological Richness**: 4.2/5. Excellent. This environment ticks all five boxes: SEEKING (food), FEAR (predators), homeostasis (hunger/fatigue), partial observability (foresight), and some room for play. It's explicitly designed for biology-grounded RL.

**License & Maintenance Status**

- **License**: Your choice (suggest MIT for future open-source sharing).
- **Maintenance**: In-house. You control the codebase and can extend it as your research evolves (e.g., add social agents, pheromones, energy budgets).

**Windows-Specific Gotchas**

- **Rendering**: Implement using pygame or matplotlib for debug visualization (optional). Headless training requires zero rendering code.
- **Dependencies**: Standard (numpy, gymnasium, torch). No physics engine, no MuJoCo, no MSVC.
- **Development**: Pure Python. No external compilation. Very portable.

**Rationale:**

1. **Biology Coverage**: Scores 4.2/5 overall. Explicitly covers SEEKING (food collection), FEAR (predators), homeostasis (hunger/fatigue), and partial observability (exploration pressure). It's designed for your use case, not retrofitted.

5. **Windows Compatibility**: Stable.
6. **Community Validation**: Published benchmark; foresight metrics can be compared to published numbers.

## Final Notes

- **Start with Instinct Gridworld**: It's the only environment purpose-built for the research question (biology + foresight). The 16–20 hour dev cost is small compared to fitting a model to an off-the-shelf env that doesn't capture target behaviors.
- **Use Crafter as a validation step**: Once the model is stable on Instinct Gridworld, train on Crafter to validate generalization to a messier, more complex environment with published benchmarks.
- **Skip Atari and dm_control for v1**: They lack the biological drives. Atari adds visual noise without biological grounding; dm_control is deterministic but has no homeostasis or fear.
- **Windows is not a blocker**: All recommended environments are Windows-stable; the custom env is pure Python.
